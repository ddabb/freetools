## 一、最初的野心：我要做一个「超级工具箱」

### 1. 我最初的想法

2026 年初，我决定做一个工具集合小程序。当时的想法非常简单粗暴：

> **「把所有有用的工具都做进去。」**

于是我打开备忘录，列了一份清单：

```
生活工具：
  - 天气查询 ✓
  - 快递查询 ✓
  - 记账本 ✓
  - 闹钟 ✓
  - 倒计时 ✓
  - 二维码生成 ✓
  - 公众号配图 ✓

工具类：
  - 计算器 ✓
  - 秒表 ✓
  - 指南针 ✓
  - 水平仪 ✓
  - 分贝仪 ✓

学习类：
  - 英语单词本 ✓
  - 古诗词查询 ✓
  - 历史今天 ✓
  - 成语词典 ✓

...大概列了 100+ 个
```

### 2. 冷静下来后的反思

清单列完后，我问自己一个关键问题：

> **「这 100 个工具，我自己会用几个？」**

答案：**大概 10 个。**

再看第二个问题：

> **「微信生态里，哪些是微信擅长的，哪些是微信做不好的？」**

- 天气查询 → 微信已经有内置天气了
- 快递查询 → 需要实时数据，需要后端
- 记账本 → 复杂的 CRUD，需要本地数据库设计
- 闹钟 → 需要后台定时器，微信不支持
- 指南针 → 需要传感器 API，体验不稳定

**突然意识到：不是所有工具都适合做进小程序。**

## 二、第一次定位修正：什么能做，什么不能做

### 1. 我画了四个象限

我把所有想法分成了四个象限：

```
            高用户需求
               │
    ┌──────────┼──────────┐
    │ 优先做    │ 看情况    │
    │          │          │
 无 │ 日期计算  │ 天气查询  │ 有
 后 │ 二维码    │ 快递查询  │ 后
 端 │ 数独     │ AI对话    │ 端
    │ 单位换算  │          │ 依
    │          │          │ 赖
    ├──────────┼──────────┤
    │ 不做      │ 坚决不做  │
    │          │          │
    │ 闹钟     │ 社交功能  │
    │ 记账     │ 支付功能  │
    │ 指南针   │ 账号体系  │
    └──────────┼──────────┘
               │
            低用户需求
```

### 2. 每个象限的取舍逻辑

**左上（优先做）：无后端依赖 + 高用户需求**
- 这些是 FreeTools 的核心
- 纯前端就能实现，零成本
- 用户真正会用到的

**右上（看情况）：有后端依赖 + 高用户需求**
- 需要外部 API 支持
- 如果找到免费 API，可以做
- 如果需要自建服务器，暂缓

**左下（不做）：无后端依赖 + 低用户需求**
- 技术上能实现
- 但用户需求不明确
- 除非顺带做，否则不投入时间

**右下（坚决不做）：有后端依赖 + 低用户需求**
- 成本高、需求低
- 完全不划算

### 3. 这次修正砍掉了什么

从最初的 100+ 个想法，砍到了 **57 个**：

| 类别 | 砍掉的原因 | 举例 |
|------|----------|------|
| 需要后台的 | 不想搭服务器 | 天气、快递、AI对话 |
| 微信限制的 | 平台不支持 | 闹钟、后台定位 |
| 体验差的 | 小程序上做不好 | 指南针、水平仪 |
| 需求不明的 | 没人用 | 成语词典、历史今天 |

## 三、第二次定位修正：「工具箱」还是「瑞士军刀」？

### 1. 一个用户的问题

小程序上线后，有个用户反馈：

> **「你们工具好多啊，但我每次只用一两个，能不能精简一下？」**

这让我开始思考一个问题：

> **用户到底是想要「很多选择」，还是想要「快速找到我要的那个」？**

### 2. 两种模式的对比

**模式 A：工具箱（现在的做法）**
```
打开小程序
↓
看到 6 个分类，57 个工具
↓
搜索或滑动找到需要的
↓
使用
```

**模式 B：搜索框（另一种做法）**
```
打开小程序
↓
看到一个搜索框
↓
输入「日期」
↓
直接跳转到日期计算
```

### 3. 我的修正方案

**两者结合：分类浏览 + 搜索直达**

```
首页（最近使用 + 热门推荐）
    ↓
搜索框（输入关键词直达）
    ↓
分类列表（按场景浏览）
```

具体改动：
- 首页改为「最近使用 + 热门推荐」
- 搜索框提到最显眼的位置
- 分类改为「折叠面板」，默认只显示常用分类

**这次修正的核心认识：工具多不是优势，找得快才是。**

## 四、第三次定位修正：免费工具的「边界」在哪里？

### 1. 一个用户的期望

> **「能不能加个 AI 写作助手？」**
> **「能不能做个记账功能？」**
> **「能不能加个翻译功能？」**

用户需求永远满足不完。那边界在哪？

### 2. 我的边界定义

我给 FreeTools 定了三条边界：

**边界 1：零成本原则**
```
能不花一分钱就实现 → 做
需要花钱买 API → 不做
需要自建服务器 → 不做
需要第三方 SDK 收费 → 不做
```

**边界 2：纯工具原则**
```
是工具（输入 → 处理 → 输出）→ 做
是平台（需要内容生态）→ 不做
是社区（需要用户互动）→ 不做
是内容（需要持续生产）→ 不做
```

**边界 3：5 分钟原则**
```
打开到用完不超过 5 分钟 → 做
需要长流程、多步骤 → 不做
需要学习成本 → 不做
需要配置才能用 → 不做
```

### 3. 根据边界筛选工具

| 工具 | 零成本 | 纯工具 | 5分钟 | 结论 |
|------|--------|-------|-------|------|
| 日期计算器 | ✅ | ✅ | ✅ | **做** |
| JSON格式化 | ✅ | ✅ | ✅ | **做** |
| AI写作助手 | ❌ | ❌ | ✅ | **不做** |
| 记账本 | ✅ | ⚠️ | ❌ | **不做** |
| 翻译 | ❌ | ✅ | ✅ | **暂不做** |
| 数独 | ✅ | ✅ | ❌ | **做**（娱乐类例外）|

### 4. 这次修正砍掉了什么

- 砍掉了「记账本」（需要复杂的数据管理，不是 5 分钟能用完的）
- 砍掉了「翻译功能」（需要付费 API）
- 砍掉了「AI 对话」（需要后端 + API 费用）
- 保留了「数独」（虽然是长流程，但属于娱乐类，可以例外）

## 五、第四次定位修正：从「给所有人」到「给自己」

### 1. 用户画像的困惑

早期我一直在想 FreeTools 的用户画像是什么：

```
年龄：20-35岁？
职业：白领、学生、程序员？
需求：效率工具？
场景：工作、生活、学习？
```

想了一圈发现：**这个画像太大了，等于没有画像。**

### 2. 一次意外的发现

有一天，一个朋友问我：
> 「你这个小程序是不是给自己做的？感觉里面的工具都挺……个人化的。」

我突然明白了：

> **FreeTools 的用户画像，就是我自己的需求。**

### 3. 拥抱「个人工具」的定位

这个认知改变了很多决策：

**之前：**
- 「要不要加个快递查询？很多人需要」
- 「要不要加个记账本？很实用」
- 「要不要加个英语学习？学生群体大」

**之后：**
- 「我自己需要快递查询吗？不需要。那就先不加。」
- 「我自己需要记账本吗？我有 Notion。不加了。」
- 「我自己需要英语学习吗？不需要。不加了。」

**核心原则变成了：我自己不用，就不做。**

### 4. 这个定位的优势

**做「给所有人」的产品：**
- 需求分散，无法聚焦
- 每个人都觉得「差点什么」
- 改来改去都不满意

**做「给自己」的产品：**
- 需求清晰（我自己就知道要什么）
- 体验可控（我自己就是用户）
- 迭代自然（我每天都在用，自然知道哪里不好）

### 5. 「给自己做」不等于「只给自己做」

> **你自己就是目标用户之一。**

这意味着：
- 你做的工具，大概率有一群和 you 类似的人也需要
- 你的需求是「真实的」，不是「想象出来的」
- 你不用的东西，大概率你做不好

## 六、修正后的最终定位

经过四轮修正，FreeTools 的定位变成了：

```
FreeTools 是一个「零成本、无广告」的微信小程序
以「给自己用」为标准，提供高频、纯前端的实用工具
每个工具都能在 5 分钟内完成使用
```

### 四个关键词

| 关键词 | 含义 | 对应的决策 |
|--------|------|-----------|
| **零成本** | 不花一分钱 | 不搭服务器、不用付费 API |
| **无广告** | 不靠广告盈利 | 不接入广告 SDK |
| **高频** | 我每天/每周都在用 | 不做低频工具 |
| **纯前端** | 所有计算在本地完成 | 不依赖后端服务 |

### 一句话总结

> **FreeTools：我做给自己的免费工具箱，如果你刚好也需要，那很好。**

## 七、给做产品的人的建议

### 1. 定位不是「定义出来的」，是「砍出来的」

没有哪个产品一开始就清楚自己是什么。

**定位是一个不断做减法的过程：**
- 列出所有想法
- 砍掉一半
- 再砍掉一半
- 剩下的就是你的定位

### 2. 「不做」比「做」更重要

做产品最难的决策不是「做什么」，而是「不做什么」。

每个被砍掉的需求，背后都有一个理由：
- 需求不够真实
- 成本太高
- 超出能力范围
- 违背核心原则

### 3. 给产品画一个「不做清单」

> 比起「我们要做什么」，更应该清楚「我们坚决不做」。

我的不做清单：
- ❌ 不做需要注册登录的功能
- ❌ 不做需要付费 API 的功能
- ❌ 不做需要自建服务器的功能
- ❌ 不做需要长流程的功能
- ❌ 不做我自己不用的功能

### 4. 定位可以变，但要记录原因

每次修正定位后，记录下来：
- 为什么要改？
- 改了之后砍掉了什么？
- 新的定位是什么？

这些记录在复盘时会非常有价值。

---

## 八、写在最后

FreeTools 的定位，从最初的「超级工具箱」，修正为「给自己用的免费工具箱」。

看起来定位变「小」了，但实际上产品变得更**清晰**了。

**定位的精髓不在于「做得多大」，而在于「知道自己在做什么」。**

一个知道自己不做什么的团队，比一个什么都想做的团队，走得更远。

---

> 这是我做 FreeTools 过程中最有价值的思考：学会说「不」。
