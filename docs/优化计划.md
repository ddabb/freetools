## 一、项目现状总览

### 1.1 架构结构

```
主包 (主包 4.5MB)
├── pages/        主包页面（9个）：index、寻识、知识详情、分类/标签列表/详情、关于、日志
├── components/    7个通用组件
├── utils/        18个工具模块
├── config/       工具配置、分类、搜索、频率统计
└── colorui/      样式框架

分包 (总计约 6MB)
├── life/         21个页面（成语、日历、倒计时、写作、配图生成等）
├── text/         11个页面（词频、Markdown、排版、加密、表情等）
├── math/         10个页面（24点、数独、最大公约数、素数等）
├── unit/         11个页面（长度/重量/温度/面积/体积/速度等换算）
├── converter/     2个页面（JSON/XML、YAML/JSON）
├── financial/     3个页面（房贷、退休、物价对比）
├── health/        1个页面（健康计算器）
├── travel/        1个页面（旅行工具）
├── developer/     1个页面（Unix时间戳）
├── design/        1个页面（色彩转换）
├── bobing/        1个页面（掷彩）
└── food/          1个页面（今天吃什么）
```

### 1.2 技术栈

| 层级 | 技术 |
|------|------|
| 框架 | 原生微信小程序（无框架依赖）|
| 样式 | ColorUI 组件库 + 自定义 WXSS |
| 数据 | CDN (jsDelivr) + LocalStorage 缓存 |
| 工具库 | 18个工具模块（toast、modal、http、storage等）|
| 算法 | 成语接龙 DFS、数独 Dancing Links、24点暴力枚举 |
| 构建 | npm 包（eslint、prettier、typescript 类型）|

### 1.3 核心数据流

```
CDN (jsDelivr)
  ↓ 请求 + 304 + LRU Storage 缓存
CacheManager (fetchWithCache)
  ↓ 缓存键 cdn_ 前缀统一管理
LocalStorage (上限 10MB，保留 7MB)
```

## 二、问题清单

### 2.1 性能问题 🔴 高优先级

**P0-1：工具配置体积过大** ✅ 已完成
- `config/tools.js` 已改为懒加载 wrapper
- `tools-common.js` 存首页必需字段（同步加载，体积小）
- `tools-full.js` 存全部工具配置（搜索时才 require）
- 首页 `onLoad` 不再阻塞，搜索时才加载完整配置树

**P0-2：首页渲染性能**
- `utils/filters.wxs` 模块已创建（WXS 层过滤工具）
- ⚠️ 但 `index.wxml` 尚未引用该模块，搜索和分类仍在 JS 层 `setData` 计算
- 需要在 `index.wxml` 中引入 WXS 并改造模板

**P0-6：知识列表缓存优先加载** ✅ 已完成
- `knowledgelist.js` 中 `tryLoadFromCache()` 已实现
- 页面优先从缓存渲染，后台静默更新 CDN 数据
- ⚠️ 注意：2026-04-17 修复了变量赋值 bug，需确保新代码已生效

**P1-4：Storage 缓存无版本校验**
- 部分模块有版本管理（如 `changelog-loader.js`）
- 需要统一 `config/constants.js` 中的 `DATA_VERSION` 与 CDN URL 的版本参数

**P1-5：部分工具页面无分包预加载** ✅ 已完成
- `app.json` 中 `preloadRule` 已配置 5 个分包：`life`、`text`、`math`、`unit`、`converter`

### 2.2 代码质量问题 🟡 中优先级

**Q1-1：工具配置文件分片** ✅ 已完成
- `tools-common.js` + `tools-full.js` 已拆分
- `tools.js` 为懒加载 wrapper

**Q1-2：重复代码散落各页面**
- 成语相关 3 个页面（idiom-query、idiom-battle、idiom-chain）各自引入 `idiom-data-service.js`
- 工具详情弹窗、数据加载逻辑在多个页面重复实现
- `utils/index.js` 导出 60+ 个方法，过于臃肿，部分方法无人调用

**Q1-3：日志页面逻辑过于复杂** ✅ 已完成
- `utils/changelog-exporter.js`：XLSX 导出逻辑已抽离
- `utils/changelog-loader.js`：版本获取和日志加载已抽离
- `changelog.js` 已简化

**Q1-4：部分页面残留调试代码** ✅ 已完成
- `js-fix-plan.json` 记录 63 个文件全部处理完毕

### 2.3 用户体验问题 🟡 中优先级

**U1-1：工具搜索体验不足**
- 首页搜索依赖关键字匹配，不支持拼音搜索、模糊搜索、同音字
- 无搜索热词、热门推荐

**U-2：页面加载状态不统一**
- 有的页面有骨架屏/加载动画，有的只有文字"加载中..."
- 错误状态（网络失败）处理方式不统一
- 部分页面无空状态设计

**U-3：分享功能不完整** ✅ 已完成
- `app.js` 中已实现 `getToolByPath`、`createToolShare`、`createShareByPath`
- 所有页面可通过 `getApp().createShareByPath()` 获取分享配置

**U-4：标签详情页交互修复** ✅ 已完成
- 面包屑已移除，`onTagTap` 跳转标签详情

### 2.4 数据管理问题 🟡 中优先级

**D-1：CDN 数据无版本号管理**
- CDN 上的 JSON 文件（成语数据、知识库索引等）没有统一版本控制
- 数据更新后需要手动清理用户本地缓存才能生效

**D-3：成语数据压缩** ⚠️ 已移除（误报）
- 原文档称 `idiom-data-service.js` 约 **1.5MB**，实际仅 **11KB**
- 11KB 在 4G 网络下 < 50ms，CDN 加载且按需触发，不构成问题
- 无需 TSKV 压缩，此项删除

### 2.5 安全合规问题 🟢 低优先级

**S-1：邮箱/公众号硬编码在代码中**
- `about.js` 中 `1011888891@qq.com` 硬编码
- `idiom-data-service.js` 中的联系信息同样硬编码

**S-2：部分 CDN URL 未加 SRI 校验**
- 仅使用 jsDelivr，依赖单一 CDN

## 五、风险与依赖

### 5.1 技术风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| WXS 过滤性能不如预期 | 大量工具时仍卡顿 | 回退到方案 B（预计算 Map）|
| CDN 版本更新后旧缓存未清理 | 用户看到过期数据 | `DATA_VERSION` 更新时主动清缓存 |

### 5.2 资源依赖

- **GitHub Actions CI：** 需配置自动化检查工具配置格式
- **测试环境：** 需要 Staging 版本用于回归测试

### 5.3 上线策略

1. **分阶段灰度：** 优化完成后灰度 10% 用户，观察 3 天无异常再全量
2. **数据向后兼容：** 缓存版本变化时自动清除旧缓存
3. **回滚方案：** Git Tag 回滚，CDN 文件保留上一版本
