## 一、项目现状总览

### 1.1 架构结构

```
主包 (主包 4.5MB)
├── pages/        主包页面（9个）：index、寻识、知识详情、分类/标签列表/详情、关于、日志
├── components/    7个通用组件
├── utils/        18个工具模块
├── config/       工具配置、分类、搜索、频率统计
└── colorui/      样式框架

分包 (总计约 6MB)
├── life/         21个页面（成语、日历、倒计时、写作、配图生成等）
├── text/         11个页面（词频、Markdown、排版、加密、表情等）
├── math/         10个页面（24点、数独、最大公约数、素数等）
├── unit/         11个页面（长度/重量/温度/面积/体积/速度等换算）
├── converter/     2个页面（JSON/XML、YAML/JSON）
├── financial/     3个页面（房贷、退休、物价对比）
├── health/        1个页面（健康计算器）
├── travel/        1个页面（旅行工具）
├── developer/     1个页面（Unix时间戳）
├── design/        1个页面（色彩转换）
├── bobing/        1个页面（掷彩）
└── food/          1个页面（今天吃什么）
```

### 1.2 技术栈

| 层级 | 技术 |
|------|------|
| 框架 | 原生微信小程序（无框架依赖）|
| 样式 | ColorUI 组件库 + 自定义 WXSS |
| 数据 | CDN (jsDelivr) + LocalStorage 缓存 |
| 工具库 | 18个工具模块（toast、modal、http、storage等）|
| 算法 | 成语接龙 DFS、数独 Dancing Links、24点暴力枚举 |
| 构建 | npm 包（eslint、prettier、typescript 类型）|

### 1.3 核心数据流

```
CDN (jsDelivr)
  ↓ 请求 + 304 + LRU Storage 缓存
CacheManager (fetchWithCache)
  ↓ 缓存键 cdn_ 前缀统一管理
LocalStorage (上限 10MB，保留 7MB)
```

## 二、问题清单

### 2.1 性能问题 🔴 高优先级

**P0-1：工具配置体积过大**
- `config/tools.js` 包含 100+ 工具配置，每次启动全部加载
- 主包 `pages/index` 在 `onLoad` 中通过 `require()` 同步加载全部工具数据
- `allTools.map()` 每次创建新对象，内存开销大

**P0-2：首页渲染性能**
- ✅ 已处理：实现了 WXS 过滤优化
- 创建 `utils/filters.wxs` 模块，在 WXML 层进行工具过滤和排序
- 移除了 `currentCategoryTools` 和 `filteredTools` 的 `setData` 操作
- 简化了 `switchCategory` 和搜索相关方法，减少 JavaScript 计算逻辑

**P0-3：成语数据服务全量加载**
- `idiom-data-service.js` 约 5000+ 条成语数据，一次性加载到内存
- 没有按需加载（lazy loading）或分包预加载

**P1-4：Storage 缓存无版本校验**
- ✅ 已处理：添加了数据版本号管理
- `config/constants.js` 中添加 `DATA_VERSION` 常量
- `cacheManager.js` 实现版本号检查，版本变化时自动清除旧缓存
- 所有缓存操作前都会进行版本验证

**P1-5：部分工具页面无分包预加载**
- ✅ 已处理：添加了 unit、converter 分包的预加载
- `preloadRule` 现在配置了 `life`、`text`、`math`、`unit`、`converter` 五个分包的预加载
- 高频工具（换算类）现在会在首页加载时预加载，减少用户点击等待时间

### 2.2 代码质量问题 🟡 中优先级

**Q1-1：工具配置文件未分片**
- `tools.js` 单文件 1000+ 行，全部工具定义挤在一起
- 新增工具需要手动维护 `categories`、`keywords`、`description`
- 没有类型定义（TypeScript），字段无校验

**Q1-2：重复代码散落各页面**

- `utils/index.js` 导出 60+ 个方法，过于臃肿，部分方法无人调用

**Q1-3：日志页面逻辑过于复杂**
- ✅ 已处理：将 XLSX 导出逻辑和版本获取抽离为独立工具
- 创建 `utils/changelog-exporter.js`：处理 Excel 导出逻辑
- 创建 `utils/changelog-loader.js`：处理版本获取和日志加载逻辑
- 简化 `changelog.js`，使用新创建的工具模块

**Q1-4：部分页面残留调试代码**
- ✅ 已处理：清理了主要页面的调试代码
- 将 `idiom-query.js` 中的 `console.log` 替换为 `console.debug`
- 将 `cacheManager.js` 中的 `console.log` 替换为 `console.debug`
- 其他页面的调试代码将在后续版本中逐步清理

### 2.3 用户体验问题 🟡 中优先级

**U1-1：工具搜索体验不足**
- 首页搜索依赖 `tools.js` 的 `searchTools`（基于关键字匹配）
- 不支持拼音搜索、模糊搜索、同音字（如"长度"搜"cd"无效）

**U-2：页面加载状态不统一**

- 错误状态（网络失败）处理方式不统一
- 部分页面无空状态设计

**U-3：分享功能不完整**
- ✅ 已处理：创建了通用分享工具
- 创建 `utils/share.js`：实现根据页面路径自动识别工具信息的分享功能

### 2.4 数据管理问题 🟡 中优先级

**D-1：CDN 数据无版本号管理**
- CDN 上的 JSON 文件（成语数据、知识库索引等）没有版本控制
- 数据更新后需要手动清理用户本地缓存才能生效
- 建议：CDN URL 加入 `?v=YYYYMMDD` 或使用 hash 文件名

**D-2：知识库数据未分包**
- 知识库所有数据（know/ 目录约 5MB）通过 CDN 加载
- 可以考虑将常用分类（心理、健康、法律）独立 CDN 文件，按需加载

**D-3：成语数据未做压缩**
- `idiom-data-service.js` 约 1.5MB（5000 条成语）
- 可考虑使用 TSKV 格式（每行一个成语，用分隔符拼接字段）替代 JSON

### 2.5 安全合规问题 🟢 低优先级

**S-1：邮箱/公众号硬编码在代码中**
- `about.js` 中 `1011888891@qq.com` 硬编码
- `idiom-data-service.js` 中的联系信息同样硬编码
- 建议：迁移到配置文件或环境变量

**S-2：部分 CDN URL 未加 SRI 校验**
- 仅使用 jsDelivr，依赖单一 CDN
- 建议：可备选 UNPKG / GitHub Raw

## 三、优化计划

### 阶段一：性能优化（预计 2 周）

#### 3.1 首页渲染优化（P0-2）

**目标：** 首页搜索响应时间 < 100ms，分类切换无白屏

**方案 A：使用 WXS 过滤（推荐）**

将搜索和分类过滤逻辑下沉到 WXS（微信子脚本），在 WXML 层完成过滤，不触发 JS 层的 `setData`：

```
WXS 模块（filters.wxs）
  ├── filterByCategory(tools, category) → tools[]
  ├── filterBySearch(tools, keyword) → tools[]
  └── sortTools(tools) → tools[]

WXML
  <wxs module="filters" src="../../utils/filters.wxs" />
  <block wx:for="{{filters.filterByCategory(allTools, activeCategory)}}">
```

**优势：** 过滤在渲染线程完成，不触发数据传输，60fps 流畅切换。

**方案 B：预计算 + 缓存分类数据**

将分类工具预计算为 Map：

```js
// index.js 改造
this._categoryMap = {};
categories.forEach(cat => {
  this._categoryMap[cat] = allTools.filter(t => 
    t.categories.includes(cat)
  );
});
// 切换分类直接读取 Map，避免每次 filter
```

**现状已有：** `_toolFrequency` / `_toolsMap` 已部分实现，可扩展。

#### 3.2 工具配置分片（P0-1）

**目标：** 主页加载时不阻塞，工具配置按需加载

**方案：**

1. 将 `tools.js` 拆分为 `tools-common.js`（首页必加载的 10 个常用工具）和 `tools-all.js`（全部工具，仅搜索时加载）
2. 首页 `onLoad` 只加载常用工具，搜索时动态 `require()` 全部工具
3. 搜索结果命中全部工具时，后台静默预加载剩余配置

```js
// index.js
const { commonTools } = require('../../config/tools-common.js');
const _lazyTools = null;

searchTools(keyword) {
  if (!_lazyTools) {
    _lazyTools = require('../../config/tools-all.js').allTools;
  }
  // 执行搜索...
}
```

#### 3.3 成语数据按首字母懒加载（P0-3）

**目标：** 首次查询时间减少 80%

**方案：**

将成语数据按拼音首字母切分为 26 个 JSON 文件：

```
idiom/
  a.json  (阿、啊)
  b.json  (八、巴、百)
  ...  
  z.json  (紫、走、坐)
  index.json  (元数据：首字母→文件路径映射)
```

查询时：
1. 根据输入成语首字母 → 找到对应 JSON 文件 URL
2. 请求该字母文件（约 50-100KB）
3. 结果缓存到 Storage，下次直接命中

**预期效果：**
- 首次加载：按需加载 1 个字母文件 ~80KB
- 缓存命中：Storage 直接返回，0 网络请求
- 全量加载：从不加载全部 5000 条

#### 3.4 缓存版本化管理（D-1）

**方案：**

1. CDN 文件使用 `?v=YYYYMMDD` 查询参数
2. `tools-config.json` 中维护版本号
3. `cacheManager.js` 读取当前版本号，版本变化时自动清除旧缓存

```js
// config.js
const DATA_VERSION = '20260416';

// CDN URL
const IDIOM_URL = `https://cdn.jsdelivr.net/...idiom.json?v=${DATA_VERSION}`;
```

#### 3.5 工具分包预加载完善（P1-5）

**目标：** 用户点击换算类工具时无加载等待

**方案：** 更新 `preloadRule`

```json
"preloadRule": {
  "pages/index/index": {
    "network": "all",
    "packages": ["life", "text", "math", "unit", "converter"]
  }
}
```

#### 3.15 成语数据压缩（D-3）

**方案：** 将 JSON 转为 TSKV 行格式


```js
  return {
    total: info.currentSize,
    cdnCount: cdnKeys.length,
    oldestTs: getOldestTs(cdnKeys)
  };
}
```

## 五、风险与依赖

### 5.1 技术风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| WXS 过滤性能不如预期 | 大量工具时仍卡顿 | 回退到方案 B（预计算 Map）|
| 成语 TSKV 改造破坏现有数据结构 | 搜索功能失效 | 先在 Staging 环境测试，CDN 双版本共存 |
| 分包预加载增加启动流量 | 用户流量消耗 | 仅预加载用户主动点击的分类 |

### 5.2 资源依赖

- **CDN 存储：** 预计增加 50MB（新增 TSKV 文件）
- **GitHub Actions CI：** 需配置自动化检查工具配置格式
- **测试环境：** 需要 Staging 版本用于回归测试

### 5.3 上线策略

1. **分阶段灰度：** 每阶段优化完成后，通过 `preloadRule`灰度 10% 用户，观察 3 天无异常再全量
2. **数据向后兼容：** 缓存版本变化时，`cacheManager` 自动清除旧缓存，不影响用户体验
3. **回滚方案：** Git Tag 回滚，CDN 文件保留上一版本
