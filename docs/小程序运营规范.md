# 小程序运营规范

> 记录 FreeTools 微信小程序的日常运营规范、内容管理和用户反馈处理流程。

## 一、版本发布规范

### 1.1 发版前检查清单

| 序号 | 检查项 | 说明 |
|------|--------|------|
| 1 | changelog 更新 | 新增/修改/修复内容写入 `data/changelog.json` |
| 2 | tools.js 检查 | 新工具配置完整（id/name/url/categories/keywords/description） |
| 3 | app.json 注册 | 新分包已在 subPackages 中注册 |
| 4 | CDN 数据更新 | 如有数据变更，确认 CDN 文件已上传 |
| 5 | 截图准备 | App Store 截图按要求尺寸准备 |
| 6 | 内部测试 | 至少在两个设备上完整测试 |

### 1.2 版本号规范

```
主版本号.次版本号.修订号
例：2.0.1
```

- 主版本号：重大架构调整
- 次版本号：新功能上线
- 修订号：Bug 修复、小优化

### 1.3 提交审核注意事项

1. **功能描述**：必须与实际功能一致，不可夸大
2. **截图**：使用真实截图，不可 P 图
3. **类目选择**：与小程序核心功能匹配
4. **测试账号**：如需登录，附上测试账号密码

## 二、内容管理

### 2.1 知识库内容规范

知识库（随身百科）是核心功能，内容需遵循以下规范：

**Front Matter 格式**
```yaml
---
title: "系统集成项目管理工程师"
description: "软考中项备考知识库"
category: "引论"
tags: ["软考", "项目管理", "备考"]
---
```

**字段说明**

| 字段 | 必填 | 说明 |
|------|------|------|
| title | 是 | 知识标题，简洁明了 |
| description | 是 | 简要描述，不超过 100 字 |
| category | 是 | 一级分类，如"项目整合管理" |
| tags | 是 | 标签数组，用于检索，支持多个 |

### 2.2 标签命名规范

- 禁止使用特殊字符（斜杠 `/`、井号 `#` 等）
- 优先使用已有标签，减少标签碎片化
- 单个知识最多 5 个标签

## 三、用户反馈处理

### 3.1 反馈渠道

- **小程序内**：通过"关于"页面的反馈入口
- **公众号**：后台消息
- **GitHub Issue**：开源社区反馈

### 3.2 反馈分类与响应

| 类型 | 定义 | 响应时间 |
|------|------|----------|
| Bug | 功能无法正常使用 | 24h 内确认，48h 内修复 |
| 优化建议 | 功能可用但体验不好 | 1 周内评估，酌情排期 |
| 新功能需求 | 建议增加功能 | 1 个月内评估 |
| 内容错误 | 知识库内容有误 | 48h 内更正 |

### 3.3 反馈处理流程

```
收到反馈 → 分类 → 评估优先级 → 排入迭代 → 通知用户
```

## 四、数据统计

### 4.1 关键指标

| 指标 | 说明 | 目标 |
|------|------|------|
| DAU | 日活跃用户数 | 持续增长 |
| 工具使用次数 | 各工具点击量 | 头部工具 > 1000次/天 |
| 知识库访问量 | 章节/知识点访问 | 待定 |
| 留存率 | 次日/7日留存 | 次日 > 30% |

### 4.2 数据查看

通过微信公众平台后台「数据分析」查看：
- 实时访问数据
- 用户画像
- 留存分析
- 页面访问情况

## 五、公众号运营

### 5.1 内容节奏

| 频率 | 内容 |
|------|------|
| 每周 1 篇 | 功能介绍/使用技巧 |
| 每月 1 篇 | 版本更新日志 |
| 按需 | 临时活动/节日主题 |

### 5.2 文章选题方向

- 功能背后的开发故事（数独求解器、成语接龙等）
- 工具使用教程（知识库使用、Excel 导出等）
- 个人开发者经验分享
- 开源项目介绍

### 5.3 文章风格

- 真诚、不装
- 有具体案例，不空谈
- 图文并茂，界面截图
- 结尾有 Call to Action（引导使用/反馈）
