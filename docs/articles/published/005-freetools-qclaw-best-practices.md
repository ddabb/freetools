## 一、为什么用 AI 辅助开发？

- ✅ 会写 JavaScript、Python

2. 找到一个方案 → 复制粘贴
3. 出问题了 → 再搜索 → 再试

**缺点：**
- 搜索很费时间
- 找到的方案不一定最优
- 容易踩坑

**现在用 QClaw 辅助开发：**


**好处：**
- 遇到问题直接问，不用搜索

## 二、QClaw 辅助开发的最佳实践

### 实践 1：让 AI 先「读懂」你的代码

**❌ 错误做法：**

```
我：帮我优化一下代码
QClaw：好的，我来优化
（优化完了，但可能不是你要的方向）
```

**✅ 正确做法：**

```
我：先看一下这个文件，了解整体结构
QClaw：已读取文件，这是一个 xxx，功能包括 xxx
我：很好，现在帮我分析这个函数有没有性能问题
QClaw：经过分析，发现 xxx 问题，建议 xxx
```

**原因：**
- 让 AI 理解上下文，避免答非所问
- AI 可以在阅读代码时发现你忽略的问题

### 实践 2：一次只做一件事

**❌ 错误做法：**

```
我：帮我加 5 个功能，顺便修 3 个 Bug，再优化一下性能
```

**✅ 正确做法：**

```
我：第一个需求：给数独求解器加一个「清空」按钮
QClaw：好的，已完成
我：第二个需求：加一个「撤销」功能
QClaw：好的，已完成
...依次类推
```

**原因：**
- 一次做太多事容易出错
- 方便定位问题
- 提交记录更清晰

### 实践 3：让 AI 主动发现问题

**❌ 错误做法：**

```
我：写完了，你帮我检查一下
（发现问题了，改）
我：还有问题吗？
（又发现问题了，再改）
...
```

**✅ 正确做法：**

```
我：这个功能开发完成了，请你帮我全面检查一下
1. 是否有潜在 Bug？
2. 是否有边界情况没处理？
3. 是否有安全隐患？
4. 性能是否有问题？
QClaw：检查完成，发现以下问题...
```

**原因：**
- AI 可以从多个角度审视代码
- 主动发现问题比被动修 Bug 更高效

### 实践 4：保留对话历史

**做法：**
- 每次开发对话保留在 QClaw 的 session 中
- 记录关键决策和实现方案
- 方便回溯：「当时为什么这么设计？」

**工具：**
- QClaw 的 `memory/` 文件夹记录重要信息
- `MEMORY.md` 记录长期记忆
- `memory/2026-03-29.md` 记录今日工作

### 实践 5：让 AI 写测试用例

**小程序 Bug 高发场景：**

| 场景 | 容易出的 Bug |
|------|-------------|
| 表单提交 | 空值判断、类型转换 |
| 循环渲染 | 变量作用域、索引错乱 |
| 异步请求 | 加载状态、错误处理 |
| 数据存储 | 格式错误、容量限制 |

**让 AI 帮你想测试用例：**

```
我：这个函数需要处理哪些边界情况？
QClaw：需要处理：
1. 空字符串
2. null/undefined
3. 非法字符
4. 超长输入
5. 特殊符号
```

## 三、小程序开发的实战经验

### 经验 1：wx:for 循环变量作用域问题

**问题描述：**
在微信小程序中，使用双层 wx:for 循环时，内层循环会覆盖外层的变量名。

**我踩的坑：**

```html
<!-- ❌ 错误写法 -->
<view wx:for="{{board}}" wx:for-index="index">
  <view wx:for="{{item}}" wx-for-index="itemIndex">
    <view wx:for="{{[1,2,3]}}" bindtap="tap" data-row="{{index}}">
      <!-- 这里的 index 已经是内层的 0/1/2 了！ -->
    </view>
  </view>
</view>
```

**QClaw 给的解决方案：**

```html
<!-- ✅ 正确写法 -->
<view wx:for="{{board}}" wx:for-index="rowIndex" wx:for-item="row">
  <view wx:for="{{row}}" wx:for-index="colIndex" wx:for-item="cell">
    <view wx:for="{{[1,2,3]}}" wx:for-index="numIndex" bindtap="tap" data-row="{{rowIndex}}">
      <!-- 使用明确的变量名，不被覆盖 -->
    </view>
  </view>
</view>
```

### 经验 2：无后端架构设计

**小程序不适合有后端的原因：**
1. 服务器成本
2. 运维复杂度
3. 数据安全问题

**QClaw 建议的架构：**

```
┌─────────────────────────────────┐
│         用户手机                 │
│  ┌───────────────────────────┐  │
│  │      小程序前端             │  │
│  │   所有计算本地完成          │  │
│  └───────────────────────────┘  │
│              ↓                  │
│  ┌───────────────────────────┐  │
│  │      本地存储              │  │
│  │  (wx.getStorageSync)      │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│         CDN (静态数据)           │
│  jsDelivr + GitHub              │
│  (版本数据、预设内容)            │
└─────────────────────────────────┘
```

**好处：**
- 零服务器成本
- 零运维成本
- 数据隐私安全

### 经验 3：CDN 数据缓存策略

**QClaw 帮我设计的缓存机制：**

```javascript
// 伪代码
const CACHE_EXPIRE = 24 * 60 * 60 * 1000; // 24小时

function loadData() {
  // 1. 检查缓存
  const cached = wx.getStorageSync('data_key');
  const timestamp = wx.getStorageSync('data_timestamp');
  
  // 2. 缓存有效，直接用
  if (cached && Date.now() - timestamp < CACHE_EXPIRE) {
    return cached;
  }
  
  // 3. 缓存失效，从 CDN 加载
  wx.request({
    url: 'https://cdn.jsdelivr.net/.../data.json',
    success: (res) => {
      wx.setStorageSync('data_key', res.data);
      wx.setStorageSync('data_timestamp', Date.now());
    }
  });
}
```

### 经验 4：分包加载优化

**小程序主包有 2MB 限制**

```
  ↓
```

**好处：**

- 按需加载，节省流量

## 六、注意事项

### 1. AI 不是完美的

- AI 可能写出有 Bug 的代码
- AI 可能忽略边界情况
- **最终还是要自己测试**

### 2. 保持批判性思维

- AI 的建议不一定最优
- 要问「为什么这样做」
- 要考虑「有没有更好的方案」

### 3. 安全意识

- AI 不知道你的业务场景
- 不要让 AI 处理敏感数据
- 重要逻辑要自己审查

---

## 七、写在最后

**用 AI 辅助开发，不是「让 AI 写代码，我什么都不管」。**

**而是「AI + 我 = 更好的开发效率」。**

AI 的优势：
- 速度快、知识广
- 不知疲倦
- 可以并行处理多个任务

我的优势：
- 懂业务、懂用户
- 能做决策
- 能判断 AI 的输出是否正确

**最好的方式是：**
- 让 AI 做它擅长的（写代码、查资料、找 Bug）
- 让我做我擅长的（产品决策、架构设计、质量把控）

**AI 是工具，不是替代品。**
