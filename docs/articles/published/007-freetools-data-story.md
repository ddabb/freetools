# 57 个工具的数据说话：谁在使用我的小程序？

> 没有数据的产品经理，就像没有仪表盘的飞行员。

---

## 一、我以为我知道，但数据告诉我我错了

### 1. 开发者的傲慢

做 FreeTools 初期，我对自己的「产品感觉」很自信：

> 「日期计算肯定是最常用的」
> 「数独是休闲利器，肯定有人玩」
> 「二维码生成是刚需」
> 「开发工具程序员肯定喜欢」

我的排序：
1. 开发工具
2. 数学工具
3. 生活工具
4. 文本工具

### 2. 真实数据打脸

上线一个月后，我看到了真实数据：

| 我以为排名 | 工具 | 实际排名 | 实际占比 |
|-----------|------|---------|---------|
| 2 | 日期计算 | 1 | 28% |
| 3 | 二维码生成 | 2 | 22% |
| 5 | 随机选择 | 3 | 15% |
| 7 | 字数统计 | 4 | 12% |
| 4 | 数独求解 | 5 | 8% |
| 1 | JSON格式化 | 11 | 1.2% |
| 1 | 正则测试 | 18 | 0.3% |
| 1 | YAML转换 | 35 | 0.1% |

**开发工具，我以为是最重要的，实际上使用量垫底。**

---

## 二、数据揭示的 5 个真相

### 真相 1：生活工具才是刚需

前 5 个最常用的工具，全部是生活类：

```
1. 日期计算 - 生活必需
2. 二维码生成 - 生活必需
3. 随机选择 - 生活必需
4. 字数统计 - 写作必备
5. 数独求解 - 休闲娱乐
```

**用户来小程序，是为了解决生活问题，不是来学习的。**

### 真相 2：程序员不是目标用户

我一直以为「开发工具」会是我的招牌功能：

- JSON 格式化
- 正则表达式测试
- YAML/JSON 转换
- 时间戳转换

**但这些工具的使用量，加起来不到 5%。**

原因：
- 程序员有自己的专业工具
- 手机上不方便写代码
- 这个场景本来就不成立

### 真相 3：80/20 法则在这里也适用

| 工具数量 | 使用占比 |
|---------|---------|
| 前 10 个工具 | 73% |
| 中间 20 个工具 | 22% |
| 后 27 个工具 | 5% |

**80/20 法则在产品世界里无处不在。**

### 真相 4：用户比想象中更懒

看用户行为数据：

- 平均每个用户只用过 **3.2 个** 工具
- 超过 5 个工具的用户，只有 **12%**
- 有 **40% 的用户**，只用过 1 个工具

**大多数用户，就是「用完就走」。**

### 真相 5：分享转化率超出预期

让我惊讶的数据：

| 指标 | 数值 |
|------|------|
| 点击「分享给朋友」的用户 | 8.3% |
| 通过分享进入的新用户 | 15% |
| 分享带来的用户占比 | 23% |

**每 10 个用户里，有 2 个是朋友分享带来的。**

## 三、数据改变的产品决策

### 决策 1：砍掉开发工具包

原来我规划了「专业开发工具」系列：
- JSON 格式化
- 正则测试
- 代码高亮

**现在：全部下架。**

这不是用户需要的。

### 决策 2：首页突出生活工具

**原来：** 分类页面，开发工具排在第一位

**现在：** 首页直接显示最常用的 5 个工具

降低用户找到「需要的东西」的成本。

### 决策 3：新增「每日数独」

数独排名第 5，说明有稳定的需求。

**新增功能：**
- 每日一题
- 全球同题
- 显示难度

**用户反馈出奇的好。**

### 决策 4：砍掉排行榜功能

我曾经想做「热门工具排行榜」：

- 让用户看到其他人用什么
- 增加工具发现的可能性
- 提高冷门工具的使用率

**数据告诉我：用户不会主动探索，只会搜索。**

## 四、数据教会我的 4 件事

### 1. 相信数据，不要相信直觉

我的产品直觉 vs 真实数据：

| 直觉 | 数据 |
|------|------|
| 开发工具很重要 | 开发工具几乎没人用 |
| 用户会探索新工具 | 用户只会搜索 |
| 工具越多越好 | 80%的工具没人用 |
| 用户会分享好工具 | 只有8%会分享 |

**直觉会骗你，数据不会。**

### 2. 用户说的 ≠ 用户做的

用户调研：

> 「这个功能很好用！」
> 「希望能加个XX功能！」
> 「希望能加个YY功能！」

实际行为：

- 用户说「很好用」的工具，使用率 2%
- 用户说要的功能，上线后使用率 0.5%

**用户说的，可能是他「以为」自己会用，而不是他「真的」会用。**

### 3. 沉默的大多数不会告诉你

给我提反馈的用户：
- 积极反馈：15人
- 消极反馈：3人

实际数据：
- 非常满意：无法量化
- 一般满意：无法量化
- 不满意但没说的：无法量化

**大多数不满意的用户，沉默离开了。**

### 4. 数据是冰冷的，但结论要有温度

数据说：
- 57 个工具，只有 5 个常用
- 23% 的用户是分享带来的
- 用户平均只用 3.2 个工具

但产品决策不能只看数字：

- 不能因为某个工具没人用就下架（万一有人需要呢）
- 不能因为分享转化高就强迫用户分享
- 不能因为用户只用 3 个就只做 3 个

**数据是参考，不是命令。**

## 五、我的数据复盘方法

### 每周必看的 3 个指标

```
1. DAU（日活）：有多少人在用
2. Top 10 工具：哪些功能最受欢迎
3. 分享转化：用户愿意分享吗
```

### 每月必做的 1 件事

```
淘汰分析：
- 哪些工具使用率低于 1%？
- 原因是什么？
- 值得保留吗？
```

### 每个版本必做的 1 件事

```
对比分析：
- 这个版本新增的工具，用的人多吗？
- 这个版本优化的功能，有正向提升吗？
- 没有的话，说明方向错了
```

## 六、写在最后

**数据不会说谎，但需要你会解读。**

- 57 个工具，用户真正需要的只有 5 个
- 我以为重要的，用户不一定觉得重要
- 沉默的大多数，才是真正的用户

**产品经理最重要的能力，不是「创造」，而是「倾听」。**

倾听用户的行为，比倾听用户的声音更重要。
