# 我如何用「零成本」开发了一个有 57+ 工具的小程序

> 这是一个关于产品思考、技术选型、持续迭代的开发复盘。

---

## 一、从一个「痛点」开始

### 1. 我遇到的问题

作为一个产品经理，我每天都在和各种工具打交道：
- 算日期：Excel算、计算器算、手机算
- 转格式：JSON转XML、YAML转JSON
- 生成码：二维码、条形码
- 做图片：封面图、分享图

**每个需求都要找网站，每个网站都要看广告。**

我开始思考：**能不能有一个地方，把这些小工具都集合在一起？**

### 2. 我的解决方案

```
问题：需要很多小工具，但没有好用的集合
解决方案：做一个工具集合小程序
```

但我不是为了做小程序而做小程序，我的核心诉求是：
1. **要免费**：我自己也要用，不能收费
2. **要无广告**：受够了每个工具网站都弹广告
3. **要快**：打开就用，不要注册登录
4. **要好用**：界面要好看，操作要流畅

## 二、技术选型：为什么选择「无后端」架构？

### 1. 传统的 B/S 架构问题

```
用户 → 前端界面 → 后端服务器 → 数据库
         ↑__________________↓
              API 调用
```

**问题：**
- 服务器成本：最便宜的云服务器也要 30 元/月
- 运维成本：需要监控、更新、安全防护
- 数据安全：用户数据存在服务器上，有泄露风险

### 2. 我的「无后端」架构

```
用户 → 小程序前端 → 本地计算（JavaScript）
                   ↓
              CDN（仅用于静态数据）
```

**优势：**
- **零服务器成本**：所有计算都在用户手机上完成
- **零运维成本**：不需要监控、不需要更新服务器
- **隐私安全**：用户数据只存在自己手机上

### 3. CDN 在这个架构里的角色

CDN（内容分发网络）在这个架构里扮演了「数据分发」的角色：

```
我的数据文件（JSON）
    ↓
GitHub 仓库
    ↓
jsDelivr CDN（免费）
    ↓
小程序请求
```

**使用场景：**
- 版本更新日志
- 预设题目（数独、24点）
- 每日一言等静态数据

**为什么不用 CDN 存储用户数据？**
- 用户数据应该存在用户本地（手机）
- 不应该把用户数据存到服务器

## 三、开发思路：如何组织 57+ 工具？

### 1. 分包策略

微信小程序有「分包」机制，可以把代码分成多个包：

```
主包（基础框架）
    ↓
生活工具包（12个工具）
    ↓
数学工具包（9个工具）
    ↓
文本工具包（9个工具）
    ↓
...
```

**好处：**
- 用户首次打开只下载主包，速度快
- 用到哪个工具才下载对应的包
- 节省流量，加快加载速度

### 2. 工具分类设计

我按照「使用场景」分类，而不是「技术类型」分类：

| 分类 | 分类依据 | 举例 |
|------|---------|------|
| 生活工具 | 日常生活场景 | 日历、二维码、日期计算 |
| 数学工具 | 需要数学计算 | 数独、24点、质数判断 |
| 文本工具 | 文本处理 | 字数统计、Markdown、密码生成 |
| 开发工具 | 程序员场景 | JSON转换、时间戳、颜色转换 |

**为什么这样分？**

因为用户想的是「我今天要用什么工具」，而不是「我要学什么技术」。

### 3. 工具卡片设计

每个工具用一个卡片展示：

```
┌─────────────────────────┐
│ 🧩 数独求解器            │  ← 图标 + 名称
│ 快速解决数独难题        │  ← 简短描述
│                        │
│ 使用次数: 1,258        │  ← 使用频率（可选）
└─────────────────────────┘
```

**设计原则：**
- 图标要直观：一看到就知道是什么
- 名称要简洁：5个字以内说清楚
- 描述要准确：让人一眼知道用途

## 四、核心功能的技术实现

### 1. 数独求解器

**算法：回溯法 + 约束传播**

```javascript
// 简化版思路
function solveSudoku(board) {
  // 1. 找到空格
  const empty = findEmpty(board);
  
  // 2. 如果没有空格，说明填完了
  if (!empty) return true;
  
  // 3. 尝试填入数字
  for (let num = 1; num <= 9; num++) {
    if (isValid(board, empty, num)) {
      board[empty] = num;
      
      // 4. 递归求解
      if (solveSudoku(board)) return true;
      
      // 5. 回溯
      board[empty] = 0;
    }
  }
  
  return false;
}
```

**优化：约束传播**
- 先填「候选数最少」的格子
- 大部分数独可以在填 20-30 个格子后大幅加速

## 五、数据更新机制

### 1. 为什么需要更新机制？

有些数据需要更新：
- 版本日志（每次更新都要改）
- 每日一言（每天更新）
- 预设题目（可能扩充）

**但我不想搭建服务器……**

### 2. CDN + GitHub 方案

```
我修改 JSON 文件
    ↓
push 到 GitHub
    ↓
jsDelivr 自动同步（5分钟）
    ↓
小程序请求获取最新数据
    ↓
缓存到用户手机
```

**实现代码：**

```javascript
// 加载CDN数据
function loadData() {
  // 1. 检查缓存
  const cached = wx.getStorageSync('data_key');
  if (cached && !isExpired()) {
    return cached;
  }
  
  // 2. 请求CDN
  wx.request({
    url: 'https://cdn.jsdelivr.net/gh/xxx/docs/data/xxx.json',
    success: (res) => {
      // 3. 缓存到本地
      wx.setStorageSync('data_key', res.data);
      // 4. 更新缓存时间戳
      wx.setStorageSync('data_timestamp', Date.now());
    }
  });
}
```

### 3. 缓存策略

| 数据类型 | 缓存时间 | 更新频率 |
|---------|---------|---------|
| 版本日志 | 24小时 | 每次发版 |
| 每日一言 | 1小时 | 每天 |
| 预设题目 | 30天 | 偶尔更新 |
| 用户设置 | 永久 | 随时 |

## 七、关键决策复盘

### ✅ 正确的决策

1. **选择无后端架构**
   - 省去了服务器成本和运维成本
   - 数据隐私更安全

2. **用 CDN 管理动态数据**
   - 不需要服务器就能更新数据
   - 免费、全球加速

3. **按场景分类工具**
   - 用户更容易找到需要的工具
   - 符合用户的心智模型

### ⚠️ 可以改进的

1. **早期缺少用户反馈机制**
   - 应该早点加上「意见反馈」入口
   - 收集用户真实需求

2. **文档不够完善**
   - 很多开发决策没有记录
   - 后来的人可能看不懂
